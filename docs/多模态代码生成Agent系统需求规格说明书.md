# 多模态代码生成 Agent 系统需求规格说明书

版本：V1.0  
日期：2026-06-22  
适用项目：Multimodal-AI-Coding-Agent-Team  
核心文件：`ai_coding_agent_bailian.py`  

## 1. 项目背景

在编程学习、算法刷题、课程实训和软件工程答辩场景中，用户经常需要根据一段编程题文字或题目截图，快速获得题意理解、解题思路、可运行代码、测试用例和执行结果。传统做法通常依赖人工读题、手写代码、手动构造测试、手动运行和整理报告，过程耗时且不易标准化。

随着大模型、多模态识别和 Agent 工作流技术的发展，可以将“读题、规划、测试、写代码、执行调试、生成报告”拆分为多个职责明确的 Agent，通过大模型 API 与本地执行环境协同完成编程题求解。本项目基于阿里云百炼 OpenAI 兼容接口、Streamlit 页面、本地 Python 执行环境和轻量级 RAG 算法模板库，搭建一套多模态代码生成 Agent 系统。

本系统当前阶段以 `ai_coding_agent_bailian.py` 单文件可运行版为核心，支持文本题目输入、题目截图上传、5 个 Agent 工作流、RAG 算法模板检索、测试计划生成、Python 代码生成、本地执行、失败自动调试、项目报告下载和 API 失败兜底输出。后续可升级为 Vue3 前端 + FastAPI/Flask 后端 + SQLite 数据库 + Docker Compose 部署的前后端分离系统。

## 2. 项目目标

本项目目标是实现一个面向编程题场景的多模态代码生成 Agent 系统，使用户可以通过浏览器页面输入编程题文本或上传题目截图，系统自动完成题意识别、RAG 模板检索、解题规划、测试生成、代码生成、代码执行调试和项目文档生成。

具体目标如下：

1. 支持文本题目和图片题目两类输入方式。
2. 使用阿里云百炼文本模型和视觉模型完成大模型推理。
3. 将完整解题流程拆分为 5 个 Agent，并记录每个 Agent 的输入、输出、状态和耗时。
4. 建设轻量级 RAG 算法模板库，为解题规划和代码生成提供结构化参考。
5. 在代码生成前先生成测试计划，使代码生成受测试约束。
6. 自动提取模型返回的 Python 代码并进行本地执行。
7. 代码执行失败时调用调试修复 Agent，最多自动修复 3 轮。
8. 在无 API Key、接口失败、模型输出异常等情况下启用离线兜底，保证系统有输出。
9. 提供 Streamlit 页面展示和命令行测试入口。
10. 支持下载 Python 解法、Markdown 项目报告和完整 JSON 结果。

## 3. 项目范围

### 3.1 当前阶段范围

当前阶段为“单文件可运行演示版 + 前后端分离升级准备”。必须完成并可验收的范围包括：

1. `ai_coding_agent_bailian.py` 可通过 Streamlit 启动。
2. 文本题目输入可触发完整 Agent 工作流。
3. 图片上传可调用视觉模型进行题意识别；识别失败时可使用文本补充或默认题目兜底。
4. 系统可展示 5 个 Agent 的步骤记录。
5. 系统可展示 RAG 检索模板、测试计划、解题说明、Python 代码、执行结果和项目文档。
6. 系统可在无 API Key 情况下生成兜底代码并完成本地执行。
7. 系统可下载 Python 解法文件、项目报告和完整 JSON 结果。

### 3.2 第二阶段升级范围

第二阶段为“前后端分离工程化版本”。升级范围包括：

1. Vue3 前端页面与 FastAPI/Flask 后端接口分离。
2. 后端将 `solve_problem()` 封装为 REST API。
3. 使用 SQLite/MySQL 保存任务历史、Agent 步骤、测试用例和执行日志。
4. 前端提供任务创建、状态轮询、结果展示、历史记录、指标看板和报告下载。
5. Docker Compose 一键启动前端、后端和必要运行环境。

### 3.3 不在当前阶段范围内

以下内容不作为当前阶段强制验收范围：

1. 本地训练大模型或视觉模型。
2. 复杂向量数据库 RAG，例如 PDF 文档 embedding、FAISS/Chroma 向量召回。
3. 多用户登录、权限管理和生产级鉴权。
4. 云服务器生产部署和公网域名配置。
5. 完整在线评测系统和大规模 Benchmark 自动评分平台。

## 4. 用户角色与使用场景

### 4.1 用户角色

| 角色 | 说明 | 主要操作 |
|---|---|---|
| 普通用户 | 编程学习者、课程实训学生、答辩演示人员 | 输入题目、上传截图、查看代码和报告 |
| 项目成员 C | Agent/RAG 负责同学 | 维护 5 个 Agent、RAG 模板库、测试生成和调试修复逻辑 |
| 后端开发者 | 第二阶段接口负责同学 | 将 `solve_problem()` 封装为 REST API，保存任务历史 |
| 前端开发者 | Vue3 页面负责同学 | 调用后端接口，展示任务状态、结果和指标 |
| 指导老师/评委 | 项目验收人员 | 查看系统演示、文档、接口、运行结果和稳定性 |

### 4.2 典型使用场景

1. 用户输入“两数之和”题目，系统生成哈希表解法、测试计划、Python 代码并运行自测。
2. 用户上传编程题截图，系统调用视觉模型识别题面，再完成解题流程。
3. 用户没有配置 API Key，系统自动使用离线兜底逻辑，仍能输出代码、执行结果和报告。
4. 模型生成的代码运行失败，系统进入执行调试 Agent，尝试自动修复并记录修复过程。
5. 答辩时演示 Agent 步骤页签，说明每个 Agent 的职责、输入、输出和耗时。

## 5. 系统整体架构

### 5.1 当前阶段架构

当前阶段采用 Streamlit 单文件架构：

```text
用户浏览器
   ↓
Streamlit 页面 ai_coding_agent_bailian.py
   ↓
solve_problem() 主工作流
   ↓
5 个 Agent + RAG 模板检索 + 本地执行调试
   ↓
页面展示结果 / 下载报告 / 下载 JSON
```

当前阶段 Streamlit 同时承担前端页面、后端业务逻辑和结果展示职责，因此不依赖 Postman 才能运行。Postman 只在第二阶段后端接口测试时作为辅助工具使用。

### 5.2 第二阶段目标架构

第二阶段采用前后端分离架构：

```text
Vue3 前端
   ↓ HTTP/JSON
FastAPI 或 Flask 后端
   ↓
ai_coding_agent_bailian.py / solve_problem()
   ↓
阿里云百炼 API + 本地执行沙盒 + SQLite/MySQL
   ↓
后端返回 JSON / Markdown 报告
   ↓
前端展示结果、历史记录和指标
```

### 5.3 核心数据流

1. 用户输入题目文本或上传题目截图。
2. 系统进入题目识别 Agent，得到结构化题面。
3. 系统根据题面关键词检索 RAG 算法模板。
4. 解题规划 Agent 根据题面和模板生成解题计划。
5. 测试生成 Agent 根据题目和计划生成测试策略。
6. 代码生成 Agent 根据题目、计划、RAG 模板和测试计划生成 Python 代码。
7. 执行调试 Agent 运行代码并判断是否需要自动修复。
8. 系统生成项目报告、指标和完整 JSON 结果。
9. 页面展示并提供下载。

## 6. 功能性需求

### 6.1 文本题目输入功能

需求编号：FR-01  
需求名称：文本编程题输入  
优先级：高  

系统应提供文本输入框，允许用户输入编程题题面、补充说明或函数签名。

输入：用户输入的编程题文本。  
处理：系统将文本作为题目识别 Agent 的输入。  
输出：结构化题面和后续 Agent 工作流结果。  
验收标准：用户输入“两数之和”题目后，系统能够生成题目识别结果、RAG 模板、测试计划、代码和执行报告。

### 6.2 图片题目识别功能

需求编号：FR-02  
需求名称：编程题截图上传识别  
优先级：高  

系统应支持上传 png、jpg、jpeg、webp 格式的题目截图，并调用视觉模型识别截图中的题目内容。

输入：图片文件和可选补充文本。  
处理：系统将图片转为 data URL，调用百炼视觉模型提取题目名称、题目描述、输入输出格式、样例和数据范围。  
输出：结构化题面文本。  
异常处理：图片识别失败时，如果用户提供了补充文本，则使用补充文本继续；如果没有补充文本，则使用默认题目兜底。  
验收标准：上传清晰题目截图后，题目识别页签能展示识别出的题目内容。

### 6.3 百炼 API 配置功能

需求编号：FR-03  
需求名称：模型接口配置  
优先级：高  

系统应支持配置阿里云百炼 API Key、接口地址、文本模型、视觉模型、请求超时时间和执行超时时间。

输入：API Key、base_url、text_model、vision_model、request_timeout、execution_timeout。  
处理：系统通过 `AgentConfig` 统一管理配置。  
输出：后续所有 Agent 使用同一配置调用模型。  
验收标准：用户可在页面侧边栏输入 API Key；未输入 API Key 时系统不崩溃，并自动进入兜底模式。

### 6.4 题目识别 Agent

需求编号：FR-04  
需求名称：题目识别 Agent  
优先级：高  

系统应实现题目识别 Agent，负责接收文本或图片输入并输出结构化题面。

输入：文本题目、图片字节、图片类型。  
处理：文本输入直接整理；图片输入调用视觉模型识别；识别失败时执行兜底。  
输出：题面文本、AgentStep 记录。  
验收标准：`agent_steps` 中必须包含“题目识别 Agent”，并记录输入摘要、输出摘要、状态和耗时。

### 6.5 轻量 RAG 算法模板检索功能

需求编号：FR-05  
需求名称：RAG 算法模板检索  
优先级：高  

系统应内置轻量级算法模板库，根据题目关键词检索相关模板，并将模板提供给后续 Agent。

模板范围包括：哈希表、双指针、滑动窗口、动态规划、二分查找、图搜索、栈与队列、贪心、文本情感规则。  
输入：题面文本。  
处理：`retrieve_algorithm_templates(problem, top_k=3)` 根据关键词命中数量排序。  
输出：模板名称、描述、算法套路、命中关键词、分数。  
验收标准：输入“两数之和”时应能检索到“哈希表 / Hash Table”模板。

### 6.6 解题规划 Agent

需求编号：FR-06  
需求名称：解题规划 Agent  
优先级：高  

系统应实现解题规划 Agent，根据题目和 RAG 模板生成解题计划。

输入：题面文本、RAG 模板。  
处理：调用百炼文本模型生成题型判断、可用算法模板、解题步骤、边界条件和复杂度目标。  
输出：Markdown 格式解题规划和 AgentStep 记录。  
异常处理：API 调用失败时使用 `offline_plan_solution()` 生成兜底规划。  
验收标准：`agent_steps` 中必须包含“解题规划 Agent”，输出中应包含算法选择或边界条件说明。

### 6.7 测试生成 Agent

需求编号：FR-07  
需求名称：测试生成 Agent  
优先级：高  

系统应在代码生成之前运行测试生成 Agent，生成测试策略、基础样例、边界样例和异常样例，并将测试计划作为代码生成约束。

输入：题面文本、解题规划。  
处理：调用百炼文本模型生成测试策略；API 不可用时调用 `offline_test_plan()`。  
输出：测试计划、结构化测试用例、AgentStep 记录。  
验收标准：`agent_steps` 顺序必须为“题目识别 Agent -> 解题规划 Agent -> 测试生成 Agent -> 代码生成 Agent -> 执行调试 Agent”。

### 6.8 代码生成 Agent

需求编号：FR-08  
需求名称：Python 代码生成 Agent  
优先级：高  

系统应根据题面、解题规划、RAG 模板和测试计划生成可运行 Python 单文件代码。

输入：题面文本、解题规划、RAG 模板、测试计划。  
处理：调用百炼文本模型生成 Markdown 解题说明和 Python 代码块。  
输出：题意理解、解题思路、复杂度分析、Python 代码、自测用例。  
约束：生成代码必须包含 `_run_tests()`，并在 `__main__` 中调用。  
异常处理：模型未返回代码块时，系统应调用 `ensure_code()` 使用兜底代码。  
验收标准：系统能提取 Python 代码并在“Python 代码”页签展示。

### 6.9 本地执行与安全检查功能

需求编号：FR-09  
需求名称：代码本地执行与安全检查  
优先级：高  

系统应在本地临时目录执行生成代码，并返回退出码、标准输出和错误输出。

输入：Python 代码字符串、执行超时时间。  
处理：先检查危险关键词，再写入临时 `solution.py`，使用当前 Python 解释器执行。  
危险关键词包括：`os.system`、`subprocess.`、`shutil.rmtree`、`socket.`、`requests.`、`urllib.request`、`eval(`、`exec(`、`open(`、`__import__`。  
输出：文本执行报告。  
验收标准：对“两数之和”兜底代码执行后，应返回退出码 0 和自测通过输出。

### 6.10 执行调试 Agent

需求编号：FR-10  
需求名称：失败自动调试修复  
优先级：高  

系统应在代码执行失败时进入执行调试 Agent，调用模型尝试修复代码，最多修复 3 轮。

输入：题目、失败代码、执行日志、测试计划。  
处理：调用 `repair_code_with_bailian()` 生成修复代码，每轮修复后重新执行。  
输出：最终代码、最终执行报告、修复记录。  
验收标准：当代码执行失败时，`repair_attempts` 应记录每轮修复状态、原因和执行报告；最多不超过 3 轮。

### 6.11 离线兜底功能

需求编号：FR-11  
需求名称：API 失败兜底输出  
优先级：高  

系统应在无 API Key、接口失败、模型响应解析失败、图片识别失败或代码块提取失败时启用兜底逻辑。

输入：异常信息和当前题目。  
处理：按阶段调用离线规划、离线测试计划或离线代码模板。  
输出：仍然可展示的题目、规划、测试计划、代码、执行报告和项目文档。  
验收标准：未配置 API Key 时，系统仍能完成完整工作流并下载报告。

### 6.12 页面结果展示功能

需求编号：FR-12  
需求名称：Streamlit 页面多页签展示  
优先级：高  

系统应在页面中展示以下内容：

1. 题意识别结果。
2. Agent 步骤。
3. RAG 模板。
4. 测试计划。
5. 解题说明。
6. Python 代码。
7. 执行结果。
8. 项目文档。
9. 下载入口。

验收标准：点击生成按钮后，页面应显示上述页签，并展示对应内容。

### 6.13 文件下载功能

需求编号：FR-13  
需求名称：结果文件下载  
优先级：中  

系统应支持下载以下文件：

1. `generated_solution.py`：生成的 Python 解法文件。
2. `agent_project_report.md`：项目说明与测试报告。
3. `agent_result.json`：完整运行结果 JSON。

验收标准：下载文件内容应与页面展示结果一致。

### 6.14 命令行测试功能

需求编号：FR-14  
需求名称：CLI 快速验证入口  
优先级：中  

系统应支持通过命令行运行核心 Agent 工作流。

运行方式：

```bash
python ai_coding_agent_bailian.py --cli "两数之和：给定 nums 和 target，返回两个数的下标。"
```

验收标准：命令行应输出题意、Agent 步骤、RAG 模板、测试计划、解题说明和执行结果。

### 6.15 第二阶段后端接口功能

需求编号：FR-15  
需求名称：REST API 封装  
优先级：中  

第二阶段系统应提供后端 API，将 `solve_problem()` 封装为前端可调用接口。

接口范围：

| 接口 | 方法 | 用途 |
|---|---|---|
| `/api/health` | GET | 健康检查 |
| `/api/tasks/text` | POST | 提交文本题目 |
| `/api/tasks/image` | POST | 提交图片题目 |
| `/api/tasks` | GET | 查询任务列表 |
| `/api/tasks/{task_id}` | GET | 查询任务详情 |
| `/api/tasks/{task_id}/report` | GET | 下载 Markdown 报告 |
| `/api/tasks/{task_id}/steps` | GET | 查询 Agent 步骤 |
| `/api/tasks/{task_id}/tests` | GET | 查询测试计划和测试用例 |
| `/api/metrics/summary` | GET | 查询指标看板 |
| `/api/tasks/{task_id}/rerun` | POST | 重新执行任务 |

验收标准：前端可通过 `http://127.0.0.1:8000` 调用接口并完成一次任务展示。

## 7. 非功能性需求

### 7.1 稳定性需求

1. 未配置 API Key 时系统不得崩溃。
2. 百炼接口失败时系统不得空白，应显示兜底输出。
3. 图片识别失败时应提示并继续使用文本或默认题目。
4. 模型未返回代码块时应使用兜底代码。
5. 本地执行超时时应停止代码并返回超时报告。

### 7.2 性能需求

1. 本地代码执行默认超时时间为 8 秒，页面允许配置 2 到 30 秒。
2. 模型请求默认超时时间为 60 秒，页面允许配置 10 到 180 秒。
3. 无 API Key 兜底模式下，文本题目应在数秒内完成输出。
4. 单次任务应记录总耗时、Agent 步骤数量、RAG 模板数量、修复轮次等指标。

### 7.3 可观测性需求

1. 每个 Agent 必须记录输入摘要、输出摘要、运行状态、耗时和错误信息。
2. 系统应展示 API 是否调用、兜底是否触发、总耗时、RAG 模板数量和修复轮次。
3. 项目报告应包含 Agent 步骤详情、RAG 模板、测试计划、执行报告和修复记录。

### 7.4 安全性需求

1. API Key 不应硬编码到代码中，应通过页面输入或环境变量读取。
2. 生成代码执行前应进行危险关键词检查。
3. 代码应在临时目录中执行，执行完成后自动清理。
4. 第二阶段后端不得直接把未处理的用户输入拼接进系统命令。
5. 第二阶段应通过 Docker Sandbox 或更强沙盒机制进一步隔离执行环境。

### 7.5 兼容性需求

1. 当前 Streamlit 页面应支持 Chrome、Edge 等主流浏览器。
2. Python 运行环境建议为 Python 3.8 及以上。
3. 第二阶段 Vue3 前端应支持主流 Chromium 内核浏览器。
4. 系统应在 Windows 本地开发环境中可运行。

### 7.6 可维护性需求

1. 核心业务逻辑应集中在 `solve_problem()` 及其子函数中，避免散落在页面层。
2. Agent 配置应通过 `AgentConfig` 统一管理。
3. Agent 输出应通过 `AgentStep` 统一记录。
4. 最终结果应通过 `AgentResult` 统一返回。
5. 新增算法模板时，应按 `ALGORITHM_TEMPLATES` 的结构补充 id、name、keywords、description 和 template。

### 7.7 可部署性需求

1. 当前阶段应支持本地命令快速启动 Streamlit 页面。
2. 第二阶段应提供 Dockerfile、docker-compose.yml 和 `.env.example`。
3. 后端容器应包含 FastAPI/Flask 运行依赖。
4. 前端容器应可运行 Vue3/Vite 项目。

## 13. 风险与约束

### 13.1 主要风险

1. API Key 欠费、无权限或网络异常会导致在线模型不可用。
2. 视觉模型对模糊截图识别不稳定。
3. 大模型可能返回不符合要求的 Markdown 或代码块。
4. 本地执行生成代码存在安全风险。
5. 当前 Streamlit 单文件版不是严格前后端分离架构。
6. 第二阶段后端 API 仍需补齐，当前 `backend/` 目录尚不完整。

### 13.2 应对措施

1. 保留离线兜底逻辑，确保无 API Key 时也能演示。
2. 对图片识别失败场景支持用户补充文本。
3. 使用 `extract_code()` 和 `ensure_code()` 保证可执行代码存在。
4. 执行前进行危险关键词拦截，并限制执行超时。
5. 将 `solve_problem()` 设计为可复用核心函数，便于后续封装 API。
6. 在后端接口说明文档中明确字段转换要求。

## 16. 结论

本项目需求重点不在本地训练模型，而在多模态输入、Agent 工作流编排、轻量 RAG 检索、代码生成、自动测试调试和工程化展示。当前阶段已经具备可运行、可演示、可解释的核心 Agent 能力；第二阶段应重点补齐后端 API、数据库存储、前端联调和 Docker 部署，使项目从单文件演示版升级为完整前后端分离工程项目。
